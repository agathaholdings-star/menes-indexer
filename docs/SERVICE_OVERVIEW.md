# メンエスインデクサ サービス概要書

## 1. サービスコンセプト

**「"確認"で終わるサイトから、"発見"が始まるサイトへ」**

ユーザーは「セラピスト名 + 口コミ」で検索し、既存サイト（ME）にたどり着く。
しかしユーザーの本当の欲求は「この子で大丈夫か？」という確認だけではない。
**「もっと良い子がいるなら、そっちを知りたい」** という発見欲求がある。

メンエスインデクサは、この「発見」のニーズに応える。
口コミ投稿から構造化データを収集し、**「同じ系統 × 同じエリア × 高評価」のセラピストを提案（レコメンド）する**、次世代のプラットフォームを目指す。

---

## 2. 競合分析（ME）

### MEができていること（強み）
- **エリア検索**: 都道府県 → 地区 → 店舗 → セラピスト の階層が整備されている。
- **詳細な口コミ評価**: 8項目（ルックス、スタイル、会話、衣装、施術、鼠径部、密着、寛容）+ 100点満点。
- **コミュニティ機能**: 掲示板、チャット、BHリスト（過激口コミ）、感謝バッジなど。
- **豊富なコンテンツ**: 約20万人分のセラピストデータ・口コミが蓄積されている。

### MEができていないこと（弱点 = 勝機）
| MEの限界 | 具体的な問題 |
| :--- | :--- |
| **名前を知らないと探せない** | 検索は「サロン名」「セラピスト名」のテキスト入力のみ。「ギャル系で巨乳の子」といった属性検索は不可能。 |
| **他店舗へのレコメンドがない** | 店舗ページには同じ店のセラピストしか表示されない。「このエリアで同じタイプの高評価セラピスト」という提案がゼロ。 |
| **口コミデータが検索に活かされていない** | ルックス、スタイル、密着度などの評価項目は表示されるが、それで絞り込む機能がない。宝の持ち腐れ。 |
| **投稿ハードルが高い** | 自由記述の長文が前提。何を書けばいいかわからず、面倒。 |
| **有料会員の投稿動機がない** | 月額4,980円払えば全部見れるので、自ら情報発信する理由がない。情報の鮮度が落ちる。 |

### MEの構造的弱点（追随できない理由）
| 弱点 | 理由 | うちの優位性 |
| :--- | :--- | :--- |
| 口コミデータが非構造化 | 5年前の設計。今から変えられない | 最初から構造化設計 |
| 有料会員が投稿しない | お金払った瞬間に投稿する意味がなくなる | Review to Earnで投稿動機を維持 |
| 無料ユーザーは「仕方なく」投稿 | ハードルが高い長文フォーム | タップ式で楽に投稿 |
| レコメンド機能がない | データがないから作れない | 構造化データでレコメンド可能 |

## 3. メンエスインデクサの解決策

### A. 「タイプ検索」で発見体験を提供
- **機能**: 口コミ投稿時に「見た目のタイプ」「スタイル」「アピールタグ」を選択式で収集。
- **効果**: 「ギャル × 巨乳 × 渋谷エリア × 80点以上」といった横断検索が可能に。
- **体験**: ユーザーは「名前を知らなくても、好みのタイプで探せる」。

### B. レコメンドエンジン（口コミベース）
- **機能**: セラピスト詳細ページに「同じ系統でもっと評価高い子」を自動表示。
- **ポイント**: セラピスト単位ではなく **口コミ単位** でレコメンド。
  - 「この口コミが良いと思った人はこの口コミも好きかも」
  - 同じセラピストでも「良い日」と「悪い日」がある
  - より精度の高いマッチングが可能
- **体験**: ユーザーは「知らなかった最高の子」に出会える。

### C. タップ式投稿フォーム（1問1答形式）
- **機能**: Tinder的なUI。タイプ・スタイル・サービスレベル・パラメータをタップ/スライダーで選択。
- **効果**: 投稿ハードルが劇的に下がり、かつ構造化データが集まる。
- **体験**: スマホで1〜2分で投稿完了。ゲーム感覚でサクサク進む。

### D. Review to Earn（投稿割引モデル）
- **機能**: 有料会員が口コミを投稿するたびに、翌月の会費を¥1,000割引（最大¥4,000/月）。
- **効果**: 有料会員にも投稿動機が生まれ、情報の鮮度が維持される。サイト側は常に最新の口コミ（サイトの資産）を収集し続けられる。
- **体験**: 月4回投稿すれば、実質¥980でサービス利用可能。

## 4. ビジネスモデル

### 会員体系

| 会員種別 | 月額 | 閲覧権限 | 投稿特典 |
| :--- | :--- | :--- | :--- |
| **無料会員** | ¥0 | **投稿1件ごとに24時間の閲覧権限を付与** | - |
| **スタンダード会員** | ¥4,980 | 無制限 | 投稿1件ごとに翌月¥1,000割引（最大¥4,000） |
| **VIP会員** | ¥14,980 | 無制限 + 詳細フィルタ | 同上 |

### VIP会員限定機能
- **サービスレベルでのフィルタリング**: 「健全〜過激」の4段階で絞り込み可能。
  - 表現は隠語を使用（MEの「爆発リスト」参考）: SKR, HR など
  - ユーザーが最も知りたがっている「具体的なサービス内容」で検索できる。
- **裏掲示板アクセス**: よりディープな情報交換の場。

### 収益モデル
1. **サブスクリプション**: スタンダード会員・VIP会員からの月額課金。
2. **店舗課金（将来）**: 店舗ページのリッチ化、求人機能、予約システム連携など。

## 5. 勝利のサイクル

```
[Phase 1] SEO流入
    ↓ MEスクレイピングデータで「店舗+セラピスト+口コミ」ページ生成
    ↓ 検索でMEと並ぶ or 上回る（SEO力で勝てる実績あり）
    
[Phase 2] 無料ユーザーの口コミ投稿
    ↓ 流入ユーザー「口コミ見たい」
    ↓ 投稿すれば24時間見放題
    ↓ タップ式だからMEより圧倒的に楽
    ↓ うちに口コミが貯まる
    
[Phase 3] 構造化データの蓄積
    ↓ タイプ・スタイル・サービスレベルが自動で貯まる
    ↓ これはMEにない独自資産
    
[Phase 4] 発見体験の提供
    ↓ 「同じギャル系で高評価の子」をレコメンド
    ↓ MEにはできない体験
    ↓ 「ここなら発見がある」でリピート
    
[Phase 5] 有料会員の獲得
    ↓ 発見体験に価値を感じたユーザーが課金
    ↓ Review to Earnで有料会員も投稿し続ける
    ↓ 情報の鮮度が維持される
    
[最終状態] MEに勝つ
    - 同等のコンテンツ量（初期データ）
    - 投稿しやすい（タップ式）
    - 有料も投稿する（Review to Earn）
    - 発見できる（レコメンド）
    → MEに残る理由がなくなる
```

## 6. 口コミ投稿フォーム設計

### 設計思想
- **ユーザーにとって**: 楽に投稿できる（タップ式、分割質問）
- **システムにとって**: 高品質な構造化データが集まる（選択式で正規化）
- **Win-Win**: ユーザーの負担を減らしながら、レコメンドの精度を上げる

### 入力項目（詳細設計）

**Step 0: 対象選択**
- 店舗名（DBからサジェスト）
- セラピスト名（DBからサジェスト or 新規追加）

**Step 2: スタイル（必須・単一選択）**
- スレンダー
- 普通

---

**Step 3: サービス満足度（必須・単一選択）**

※ニッチな需要を拾うためのハッシュタグ。選択式にすることで検索ヒット率UP。

**見た目系**

**Step 6: 点数（必須）**

## 7. UI/UX方針

### デザイン基本方針
- **PCファースト + レスポンシブ**: MEベースの情報量を維持しつつ、スマホでも見やすく。
- **MEに慣れたユーザーが迷わない**: 基本的なページ構成（トップ → エリア → 店舗 → セラピスト）は踏襲。
- **「発見」を際立たせるUI**: タイプ検索、レコメンド欄、レーダーチャートなど独自要素を追加。

### 独自UI要素
- **「タイプから探す」グリッド**: トップページに6つのタイプカードを配置。
- **レーダーチャート**: セラピスト詳細ページにパラメータを可視化。
- **タイプ成分分析（円グラフ）**: 「ギャル80%、アイドル20%」のような分布表示。
- **レコメンド欄**: 「同じタイプでもっと高評価の子」を自動表示。
- **口コミブラー（ぼかし）**: 未投稿ユーザーには口コミ本文をぼかして表示。投稿CTAを促す。

### コミュニティ機能（最初から実装）
- **掲示板**: MEと同等の機能を提供。
- **グループチャット**: エリア別・趣向別のチャットルーム。
- **お気に入り共有**: 自分のリストを公開・共有。

## 9. 技術課題（要検討）

### 課題1: セラピスト在籍情報の鮮度

**問題:**
- MEのスクレイピングデータは「ある時点」のスナップショット
- 実際は毎日入店・退店がある
- 本家サロンをスクレイピングして最新情報を取る必要がある

**難点:**
- サロンごとにURL構造が違う（aromamore.com/cast vs aromabby.com/therapists）
- セレクタも違う
- 全サロンに専用スクレイパーを作るのは現実的じゃない

**解決案: LLMベースの汎用スクレイパー**
```
1. サロンの在籍ページURLをfetch
2. Claude APIにHTMLを渡す
3. 「セラピスト名とURLのリスト」を抽出させる
4. セレクタを事前定義する必要なし
```

### 課題2: データの名寄せ

**問題:**
- MEデータのセラピスト ↔ 本家データのセラピスト
- どうやって同一人物として紐付けるか

**解決案:**
- 既存の `therapist_url`（本家セラピストページURL）でマッチング
- 新規: DBにない → 新規登録
- 退店: DBにあるが本家に存在しない → status = "retired"

### 現在のSupabaseスキーマ

**salon_db:**
- salon_id, salon_name, area_id, area_name
- salon_url ← 本家サロンのURL
- me_salon_url, me_therapistlist_url

**therapist_db:**
- therapist_id, salon_id, area_id
- therapist_name, therapist_age, three_sizes
- therapist_url ← 本家セラピストページのURL
- me_therapist_url
- reviews_data, review_count, latest_date

## 10. まとめ：なぜMEを超えられるのか

### MEの限界
**「名前を知っている人の評判を"確認"するサイト」** でしかない。

### メンエスインデクサの強み
**「好みのタイプから知らない最高の子を"発見"するサイト」** になれる。

### 差別化の4つの柱

| # | 戦略 | 具体策 |
| :--- | :--- | :--- |
| 1 | **検索意図の上位概念を解決** | MEは「確認」で終わる。うちは「発見」まで連れて行く。 |
| 2 | **データを活かす設計** | 口コミデータを「表示」だけでなく「検索・レコメンド」に活用。 |
| 3 | **投稿者にメリットを還元** | 無料会員には24時間閲覧権、有料会員には月額割引。情報の鮮度が保たれる。 |
| 4 | **投稿ハードルを極限まで下げる** | タップ式で1分投稿。データの量と質を両立。 |

### ユーザーの本音に応える

> 「Aさんの口コミ見に来たけど、正直もっと良い子いるなら教えてほしい」
> 「この店は遠いから、近くで似た子いないかな」
> 「Aさん予約取れないから、似た子で空いてる子いないかな」

MEはこの本音に応えられない。うちは応える。
